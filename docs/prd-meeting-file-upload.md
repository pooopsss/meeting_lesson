# PRD: Загрузка файлов встречи

**Дата**: 2026-07-14
**Статус**: Draft

## Цель

Дать аутентифицированным пользователям возможность прикреплять файлы к встрече (документы, изображения, аудио, видео, текст, архивы) — загружать, видеть список, скачивать и удалять — с хранением на бэкенде и отображением в интерфейсе деталей встречи. Доступ к файлам строго ограничен владельцем встречи.

## Пользовательские сценарии

- Владелец встречи нажимает «Загрузить файл» в деталях встречи, выбирает файл, опционально вводит подпись → файл появляется в списке с иконкой, размером, автором и временем; тост «Файл загружен».
- Участник встречи открывает детали встречи → видит список всех прикреплённых файлов с кнопками «Скачать» и (для аудио/видео) встроенным превью-плеером.
- Загрузивший файл кликает «Удалить» в строке своего файла → подтверждение в `ConfirmDialog` → файл исчезает из списка и с диска; тост «Файл удалён».
- Пользователь пытается загрузить файл > 20 МБ (для документов) или > 200 МБ (для аудио/видео) → видит инлайн-ошибку «Файл превышает допустимый лимит», файл не сохраняется.
- Пользователь пытается загрузить файл с неподдерживаемым MIME (например `.exe`) → видит инлайн-ошибку «Неподдерживаемый тип файла», файл не сохраняется.
- Другой пользователь пытается через API получить/скачать/удалить чужой файл → получает `403`/`404`, не `200`.
- Неаутентифицированный пользователь обращается к любому эндпоинту файлов → получает `401`.

## В скоупе

**API (4 эндпоинта под существующим `auth` middleware):**

- `POST /api/meetings/{id}/files` — загрузка (`multipart/form-data`: `file`, опц. `label`).
- `GET /api/meetings/{id}/files` — список файлов встречи (DESC по `created_at`).
- `GET /api/meetings/{id}/files/{fileId}` — скачивание (стрим с `Content-Disposition: attachment`).
- `DELETE /api/meetings/{id}/files/{fileId}` — удаление (только загрузивший).

**Модель данных:**

- Новая таблица `meeting_files` (поля: `id`, `meeting_id` FK CASCADE, `user_id` FK CASCADE, `original_name`, `stored_name` UUID+ext, `mime_type`, `size` bytes, `label` NULL, `created_at`, `updated_at`).
- Индексы: `(meeting_id, created_at DESC)`, UNIQUE на `stored_name`.
- Новая Eloquent-модель `App\Models\MeetingFile` с `belongsTo(Meeting)` и `belongsTo(User)`.
- В `Meeting` — `files(): HasMany`; в `User` — `uploadedMeetingFiles(): HasMany`.
- Миграция `create_meeting_files_table`.

**Хранение:**

- Локальный диск Laravel (`local` disk), путь `storage/app/meetings/{meeting_id}/{stored_name}`.
- Конфиг `config/files.php` с картой `max_size_mb` по категориям и `allowed_mimes`.
- Запрет прямой отдачи `storage/app/...` через Nginx.

**UI/UX (PrimeVue, без роутера):**

- Новый компонент `MeetingFileList` в деталях встречи: `DataView` строк (иконка по MIME, имя, размер, автор, время, подпись, скачать, удалить).
- Для `audio/*` — нативный `<audio controls preload="none">`; для `video/*` — `<video controls preload="metadata">`.
- Диалог загрузки: PrimeVue `FileUpload` (mode=basic, customUpload) + опц. `InputText` подписи; `maxFileSize` подбирается по MIME выбранного файла; `ProgressBar` во время загрузки.
- Кнопка «Удалить» видна только `currentUser.id === file.user_id`; подтверждение через `ConfirmDialog`.
- Пустое состояние «Файлов пока нет» + подсказка.
- Тосты: success/error через PrimeVue `Toast`.

**Валидация файлов:**

- Категории и лимиты:

  | Категория | MIME | Лимит |
  |---|---|---|
  | Документы | `application/pdf`, `application/msword`, `application/vnd.openxmlformats-officedocument.wordprocessingml.document`, `application/vnd.ms-excel`, `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet` | 20 МБ |
  | Изображения | `image/jpeg`, `image/png`, `image/gif`, `image/webp`, `image/svg+xml` | 20 МБ |
  | Текст | `text/plain`, `text/csv`, `text/markdown` | 20 МБ |
  | Архивы | `application/zip` | 20 МБ |
  | Аудио | `audio/mpeg`, `audio/mp4`, `audio/wav`, `audio/ogg`, `audio/webm`, `audio/aac`, `audio/x-aac`, `audio/flac`, `audio/x-m4a` | 200 МБ |
  | Видео | `video/mp4`, `video/webm`, `video/ogg`, `video/quicktime`, `video/x-matroska`, `video/x-msvideo` | 200 МБ |

- MIME определяется через PHP `finfo` по содержимому, не по клиентскому заголовку.
- Санитизация `original_name`: убираются разделители пути и управляющие символы, длина ≤ 255.
- На диск сохраняется `stored_name` = UUID + расширение (не пользовательский ввод).
- Ошибки валидации → `422` с понятным сообщением по полю `file`.

**Права доступа:**

- Чтение/список/скачивание/загрузка: владелец родительской встречи.
- Удаление: только загрузивший пользователь.
- Чужая встреча или чужой файл → `403`/`404` (не `200`, не `400` — защита от перечисления id).

**Логирование:** upload/download/delete пишут строку с `meeting_id`, `user_id`, `file_id`, статусом (`ok`/`error`).

**Документация:** README обновляется — раздел «Как прикрепить файл», оговорка про локальное хранение, ссылка на конфиг лимитов.

## Не в скоупе

- Совместное редактирование файлов в реальном времени.
- Превью PDF-страниц, офисных документов и других нетривиальных форматов (только скачивание).
- Хранение в облаке (S3, GCS) — только локальный диск.
- Версионирование файлов / история изменений.
- Шеринг с незарегистрированными (внешними) пользователями.
- Антивирусная проверка, OCR, транскрибация, генерация превью-картинок.
- Chunked/multipart upload — большие видео грузятся одним HTTP-запросом в v1.
- Квоты на пользователя / встречу.
- Роли «участник встречи» — в v1 права есть только у владельца.

## Технические ограничения

- **Стек:** Lumen 10 + PHP 8.3 + PostgreSQL 16, фронт Vue 3 + PrimeVue 4 (Aura). Никаких новых сервисов.
- **Авторизация:** существующий bearer-токен через middleware `auth`; модель `UserSession` остаётся как есть.
- **Нет multipart-чанков** на бэкенде — загрузка одним запросом, лимит PHP `upload_max_filesize` и `post_max_size` должен быть ≥ 200 МБ.
- **Локальное хранение:** файлы живут в volume `backend/`; переживают `docker compose restart backend`, но теряются при удалении volume. В проде это надо явно задокументировать как ограничение.
- **Nginx** не должен отдавать `storage/app/...` напрямую — только через PHP, иначе теряется проверка прав.
- **MIME-проверка** обязана идти через `finfo` по содержимому файла, а не по клиентскому заголовку — иначе легко обойти allow-list переименованием.
- **Конкурентность:** параллельные загрузки в одну встречу не должны перезаписывать файлы на диске — отсюда UUID в `stored_name`.
- **Автотесты** используют трейт `DatabaseMigrations` (схема сбрасывается) — после прогона тестов нужно заново `php artisan migrate --force` для живого приложения.
- **Линт/формат:** PHP-CS-Fixer (бэк), Prettier (фронт) — должны проходить без замечаний.
- **Локализация:** размер в UI — человекочитаемый (КБ/МБ, 1 знак), время — в локали браузера.

## Критерии готовности

**Функциональность**

- [ ] Реализованы все 4 эндпоинта: `POST`, `GET` (список), `GET` (скачивание), `DELETE`.
- [ ] Валидация возвращает `422` с понятным сообщением для: пустого/отсутствующего файла, неподдерживаемого MIME, превышения лимита категории, длинной подписи.
- [ ] MIME определяется через `finfo`, а не по клиентскому заголовку.
- [ ] `original_name` санитизируется; на диск сохраняется UUID, не пользовательский ввод.
- [ ] Лимит категории (`audio/*` / `video/*` / остальные) выбирается по MIME, а не зашит константой.
- [ ] Удаление работает только для загрузившего; чужим — `403`.

**Безопасность**

- [ ] Все эндпоинты без токена возвращают `401`.
- [ ] Чужой файл/встреча не доступны через API (`403`/`404`, не `200`).
- [ ] Скачивание стримится через PHP; `storage/app/...` не отдаётся Nginx напрямую.
- [ ] Path-traversal (`../../etc/passwd` в имени) не приводит к чтению/записи чужих файлов.

**Данные и хранение**

- [ ] Миграция `meeting_files` накатывается и откатывается чисто.
- [ ] Файлы переживают `docker compose restart backend`.
- [ ] При ошибке БД после сохранения файла на диске файл удаляется (нет «осиротевших» файлов).
- [ ] Удаление файла убирает и запись в БД, и файл с диска.
- [ ] Конфиг `config/files.php` создан, лимиты и allow-list читаются из него (а не хардкодятся в контроллере).

**UI / UX**

- [ ] В деталях встречи есть список файлов: иконка по MIME, имя, размер, автор, время, подпись, кнопки скачать/удалить.
- [ ] Кнопка «Загрузить файл» открывает диалог с `FileUpload` и опц. подписью.
- [ ] Во время загрузки диалог блокируется и виден прогресс.
- [ ] Пустое состояние, успех и ошибка показываются тостами/инлайн-сообщениями.
- [ ] Аудио — нативный `<audio controls>`; видео — `<video controls>`.
- [ ] Кнопка «Удалить» видна только загрузившему; подтверждение через `ConfirmDialog`.
- [ ] Действующий лимит (20 / 200 МБ) отображается пользователю в диалоге загрузки.
- [ ] Список из 100 файлов рендерится без видимых лагов.

**Качество**

- [ ] Автотесты из раздела «Тестирование» зелёные в CI.
- [ ] PHP-CS-Fixer и Prettier проходят без замечаний.
- [ ] Ручной smoke-чеклист пройден на desktop и mobile-вьюпорте.
- [ ] Логи пишутся для upload/download/delete с `meeting_id`, `user_id`, `file_id`, статусом.
- [ ] README обновлён: раздел «Как прикрепить файл», оговорка про локальное хранение, ссылка на конфиг лимитов.

**Производительность**

## Глоссарий

- **Файл (attachment)** — запись в `meeting_files` + бинарный blob на диске.
- **Владелец встречи** — пользователь, создавший встречу (`meetings.user_id`).
- **Категория файла** — логическая группа MIME (`document` / `image` / `text` / `archive` / `audio` / `video`); определяется по префиксу `type/*` и задаёт лимит размера.
- **`finfo`** — PHP-расширение `fileinfo`, определяющее MIME по содержимому файла.
- **`stored_name`** — сгенерированный UUID + расширение; единственное имя, под которым файл живёт на диске.
- **`original_name`** — имя файла, заданное пользователем при загрузке; санитизируется, может дублироваться у разных файлов.
- **`label`** — опциональная человекочитаемая подпись, отображается в UI рядом с файлом.
